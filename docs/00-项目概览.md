## 1. 项目背景：为什么需要多Agent临床辅助决策系统

### 1.1 医疗AI的现实困境

传统的医疗信息系统（HIS，Hospital Information System）虽然实现了病历数字化，但在**辅助临床决策**层面仍然高度依赖医生的个人经验和记忆。一位全科医生每天面对数十位患者，需要在几分钟内完成以下工作：

1. **信息采集**：从患者口述中提取关键症状、既往病史、用药情况、过敏史
2. **鉴别诊断**：根据症状在数千种疾病中缩小范围，给出排名的可能性列表
3. **治疗方案**：选择恰当的药物，检查药物交互作用（DDI），考虑过敏史和禁忌症
4. **医学编码**：将诊断结果编码为ICD-10标准编码，用于保险报销和统计分析
5. **合规审计**：确保患者隐私数据符合HIPAA等法规要求，防止数据泄露

这五个步骤涉及**完全不同的知识领域**，且每一步都可能出错。这正是多Agent系统发挥作用的理想场景——让不同的"专家"各司其职。

> **小白解读**：你可以把这个系统想象成一个"虚拟医疗团队"。就像医院里有护士负责问诊、主治医生负责诊断、药剂师负责开药、编码员负责录入系统、审计员负责检查合规——我们用5个AI Agent分别模拟这5个角色。

### 1.2 为什么需要AI介入

| 痛点 | 具体说明 | AI可以做什么 |
|------|---------|-------------|
| 信息遗漏 | 患者口述杂乱无章，医生在时间压力下容易遗漏关键信息 | NLP（自然语言处理）解析，自动结构化提取 |
| 诊断偏差 | 罕见病容易被忽略，认知偏差（锚定效应）影响判断 | 知识图谱辅助鉴别诊断，给出多个可能性排名 |
| 药物交互 | 多药联用风险难以记忆，FDA批准的药物有数万种 | 自动DDI（Drug-Drug Interaction）检查 |
| 编码效率 | ICD-10有超过7万个编码，手动查找费时费力 | 自动编码 + DRGs分组 |
| 合规风险 | HIPAA违规罚款可达150万美元/年 | 自动PHI（受保护健康信息）检测与数据脱敏 |

### 1.3 为什么选择LLM（大语言模型）

大语言模型（如GPT-4o）在以下方面有天然优势：

- **自然语言理解**：直接处理非结构化的患者叙述文本，无需预定义模板
- **医学知识**：经过海量医学文献预训练，具备广泛的医学常识
- **推理能力**：能从症状集合推导出可能的诊断，并给出推理过程
- **格式化输出**：通过Prompt Engineering（提示工程）输出结构化JSON数据

但LLM也有明显的局限性——**幻觉（Hallucination）**，即编造不存在的事实。例如LLM可能"发明"一个不存在的ICD-10编码。这就是为什么我们需要：

1. **知识图谱（GraphRAG）**来约束和验证LLM的诊断输出
2. **内置编码库**来校验ICD-10编码的正确性
3. **规则引擎（Audit Agent）**来进行确定性的合规检查，而不依赖LLM

### 1.4 为什么是多Agent而不是单Agent

核心理念：**分而治之（Divide and Conquer）**。

把一个复杂的任务拆分成5个独立的子任务，每个子任务由一个专用Agent负责。这样做的好处：

- 每个Agent的Prompt短而精，LLM执行质量更高
- 每个Agent可以独立测试和调试
- 出错时能精确定位到具体是哪个Agent的问题
- 新增功能只需添加新Agent，不影响现有流程

## 2. 什么是Agent与多Agent系统

### 2.1 Agent的基本概念

**Agent（智能体）**是一个能够"感知环境 → 做出决策 → 执行动作"的自主实体。在软件工程的语境下，一个Agent通常由三部分组成：

```
┌─────────────────────────┐
│         Agent            │
├─────────────────────────┤
│  Prompt（角色定义）       │  ← 告诉LLM它是谁、该做什么、输出什么格式
│  LLM（大脑）             │  ← 负责理解输入、推理和生成输出
│  Tools（工具）            │  ← 可调用的外部能力（数据库查询、API调用等）
└─────────────────────────┘
```

> **小白解读**：Agent = 一个有明确职责描述的"员工" + 一个聪明的"大脑"（LLM）+ 一套可以使用的"工具箱"。

### 2.2 多Agent系统

**多Agent系统（Multi-Agent System, MAS）**就是让多个Agent协作完成一个复杂任务。每个Agent专注于自己擅长的子任务，就像医院里的多学科协作：

| 医院角色 | 对应Agent | 职责 |
|---------|----------|------|
| 问诊护士 | Intake Agent | 收集和整理患者信息 |
| 诊断医生 | Diagnosis Agent | 分析症状，做出鉴别诊断 |
| 临床药剂师 | Treatment Agent | 制定治疗方案，检查药物交互 |
| 医学编码员 | Coding Agent | 将诊断映射为ICD-10编码 |
| 合规审计官 | Audit Agent | 检查数据是否符合HIPAA规定 |

### 2.3 Agent编排模式

"编排"指的是如何协调多个Agent的执行顺序和通信方式。常见的模式有：

| 模式 | 工作方式 | 优点 | 缺点 | 适用场景 |
|------|---------|------|------|---------|
| **Pipeline（流水线）** | Agent按固定顺序依次执行 | 流程确定、可审计、好调试 | 灵活性相对低 | 流程明确的场景（如本项目） |
| **Supervisor（监督者）** | 一个中心Agent负责任务分发 | 灵活、可动态路由 | 中心节点是瓶颈 | 任务类型不确定的场景 |
| **Debate（辩论）** | 多个Agent各给意见，再综合 | 多视角、减少偏差 | 慢、成本高 | 需要多视角分析的场景 |
| **Voting（投票）** | 多个Agent独立决策，取多数 | 高可靠性 | 资源消耗大 | 关键决策场景 |

我们选择 **Pipeline模式** 是因为临床决策流程天然就是线性的：先采集信息 → 再诊断 → 再治疗 → 再编码 → 最后审计。同时，我们加入了**条件路由**：当诊断信息不足时，流程可以回退到采集环节补充信息。

```
                     ┌──── needs_more_info = true ────┐
                     │                                │
                     ▼                                │
Intake ──→ Diagnosis ──→ Treatment ──→ Coding ──→ Audit ──→ END
                     │
                     └──── needs_more_info = false → 继续往下
```

## 3. 多Agent vs 单Agent的对比

### 3.1 单Agent方案示例

```python
# 单Agent方案：一个巨大的Prompt试图完成所有事情
SINGLE_AGENT_PROMPT = """
你是一个全能医疗AI，请一次性完成以下所有任务：
1. 解析患者信息为JSON
2. 做出鉴别诊断并给出置信度
3. 给出治疗方案并检查药物交互
4. 编码为ICD-10并分组DRGs
5. 检查所有输出的HIPAA合规性
返回一个包含以上全部内容的完整JSON...
"""
# 问题：Prompt超过2000词，LLM容易"迷失"在中间部分
```

### 3.2 详细对比表

| 维度 | 单Agent | 多Agent（本项目） |
|------|---------|-----------------|
| **Prompt复杂度** | 一个超长Prompt（>2000词），LLM容易"遗忘"后面的指令 | 每个Agent一个短而精的Prompt（~300词），执行质量更高 |
| **可调试性** | 输出一个巨大JSON，出错很难定位 | 每个Agent独立输出，可以逐步检查每一步 |
| **可测试性** | 只能做端到端测试 | 每个Agent可以写独立的单元测试 |
| **可扩展性** | 加新功能需要改Prompt，可能影响全局 | 加新Agent只需注册一个节点，不影响其他Agent |
| **错误隔离** | 一步出错，整个结果不可用 | 一个Agent出错，其他Agent仍可执行（优雅降级） |
| **并行开发** | 只有一个文件，多人协作困难 | 每个Agent独立文件，团队可以并行开发 |
| **Token成本** | 长Prompt + 长Response = 高成本 | 每步Prompt和Response较短，总成本更可控 |
| **质量天花板** | ~5个子任务后质量显著下降 | 理论上可扩展到几十个Agent |

### 3.3 为什么企业级项目都选多Agent

- **Google MedPaLM**：多模块流水线，分别处理理解、推理、生成
- **Microsoft BioGPT**：将生物医学NLP拆分为多个专用模型
- **OpenAI GPTs**：允许创建多个专用GPT并让它们协作
- **LangGraph框架本身**：就是为多Agent编排而设计的

## 4. 项目目标和范围

### 4.1 项目目标

| 目标 | 说明 |
|------|------|
| 构建完整的多Agent临床辅助决策流水线 | 5个Agent依次处理，从自然语言到编码审计报告 |
| 提供三种语言实现 | Python（LangGraph）、Java（Spring Boot + LangGraph4j）、Go（Gin + 手写Pipeline） |
| 集成医疗行业标准 | FHIR R4互操作性、ICD-10编码、HIPAA合规 |
| 知识图谱增强诊断 | GraphRAG技术，基于Neo4j存储医学知识图谱 |
| 面试展示友好 | 代码结构清晰，文档齐全，可在面试中深度讲解 |

### 4.2 功能范围

**包含（In Scope）**：

- 自然语言患者信息解析 → 结构化PatientInfo（FHIR对齐）
- 基于LLM的鉴别诊断（含置信度评分和证据链）
- 循证医学治疗方案推荐（含DDI药物交互检查）
- ICD-10-CM自动编码 + MS-DRGs分组
- HIPAA Safe Harbor去标识化（18类PHI检测）+ 不可变审计日志
- FHIR R4资源映射（Patient、Condition、MedicationRequest）
- 内置知识库（症状-疾病映射、ICD-10编码库、DDI数据库）

**不包含（Out of Scope）**：

- 真实的电子病历系统（EMR）集成
- FDA认证级别的临床部署
- 实时生命体征监控
- 医学影像分析（CT、MRI等）
- 多轮交互式问诊（当前为单次输入/输出）

### 4.3 技术栈总览

| 层级 | 技术 | 用途 |
|-----|------|------|
| AI编排 | LangGraph / LangGraph4j / 手写Pipeline | Agent流水线编排与条件路由 |
| LLM | OpenAI GPT-4o-mini（默认） | 自然语言理解与结构化生成 |
| 知识图谱 | Neo4j + Cypher | GraphRAG医学知识存储与多跳查询 |
| Web框架 | FastAPI / Spring Boot 3.3 / Gin | REST API服务层 |
| 数据库 | PostgreSQL | 审计日志（WORM）+ 会话持久化 |
| 缓存 | Redis | 结果缓存与会话管理 |
| 医疗标准 | FHIR R4、ICD-10-CM、HIPAA | 行业互操作性与合规 |
| 容器化 | Docker + docker-compose | 一键启动所有基础设施依赖 |
| 数据验证 | Pydantic v2 / Jakarta Validation / gin validator | 输入输出数据校验 |
| 日志 | structlog / SLF4J / Go标准log | 结构化日志记录 |

## 5. 系统五大Agent一览

| # | Agent名称 | 角色类比 | 输入 | 输出 | 是否调用LLM | 核心能力 |
|---|----------|---------|------|------|------------|---------|
| 1 | **Intake Agent** | 问诊护士 | `raw_input`（自然语言描述） | `patient_info`（结构化JSON） | 是 | NLP解析、FHIR字段对齐、数据校验 |
| 2 | **Diagnosis Agent** | 诊断医生 | `patient_info` | `diagnosis`（鉴别诊断列表+置信度） | 是 | GraphRAG集成、证据链推理、信息不足检测 |
| 3 | **Treatment Agent** | 临床药剂师 | `patient_info` + `diagnosis` | `treatment_plan`（治疗方案） | 是 | DDI检查、禁忌症验证、循证推荐 |
| 4 | **Coding Agent** | 医学编码员 | `diagnosis` + `treatment_plan` | `coding_result`（ICD-10+DRGs） | 是 | ICD-10-CM编码、DRGs权重和平均住院日 |
| 5 | **Audit Agent** | 合规审计官 | 全部state字段 | `audit_result`（合规报告） | **否** | PHI正则检测、数据脱敏、WORM审计日志 |

> **关键设计决策**：Audit Agent 故意不使用LLM。因为合规检查需要100%确定性——LLM可能产生幻觉导致漏检PHI（受保护健康信息），而正则表达式匹配是完全确定性的。

## 6. 三种语言实现对照

本项目提供三种语言的完整实现，方便不同技术栈的开发者学习和面试：

| 维度 | Python | Java | Go |
|------|--------|------|----|
| **Web框架** | FastAPI 0.115+ | Spring Boot 3.3 | Gin 1.10 |
| **Agent编排** | LangGraph StateGraph（声明式图） | 显式循环Pipeline | for循环+break Pipeline |
| **LLM调用** | langchain-openai ChatOpenAI | Spring AI ChatClient | sashabaranov/go-openai |
| **状态模型** | Pydantic ClinicalState | Lombok @Data ClinicalState | Go struct ClinicalState |
| **FHIR服务** | ✅ fhir_service.py | 文档描述 | ✅ fhir.go |
| **GraphRAG** | ✅ graphrag_service.py | — | — |
| **HIPAA检测** | ✅ hipaa_service.py + Audit Agent | Audit Agent内联 | ✅ hipaa.go + Audit Agent |
| **ICD-10/DDI** | ✅ 独立服务模块 | Agent内联 | ✅ 独立服务模块 |
| **默认端口** | 8000 | 8080 | 8090 |
| **状态检查点** | MemorySaver（LangGraph内置） | 无 | 无 |

**选择建议**：

- **Python**：最完整的实现。适合AI/ML背景开发者，LangGraph是当前最主流的Agent编排框架
- **Java**：适合有Spring Boot经验的后端工程师。展示了企业级Java如何集成LLM
- **Go**：适合云原生/高性能方向。手写Pipeline展示了Agent编排的底层原理

## 7. 参考的企业级开源项目

本项目的设计参考了以下前沿的企业级开源项目和学术成果：

### 7.1 Medical-Graph-RAG（ACL 2025）

- **论文**：发表于自然语言处理顶级会议ACL 2025
- **核心思想**：利用知识图谱增强RAG（检索增强生成），解决传统RAG在医学领域的局限性
- **三层知识架构**：
  - 临床数据层（MIMIC-IV电子病历数据集——全球最大的公开重症监护数据库）
  - 医学文献层（PubMed超过3600万篇生物医学文献、S2ORC开放研究语料库）
  - 受控词汇层（UMLS统一医学语言系统、SNOMED-CT临床术语、ICD-10国际疾病分类）
- **U-Retrieval技术**：Top-down（从图谱全局搜索关联实体）+ Bottom-up（从局部实体出发扩展关系网络）双向检索策略
- **本项目借鉴点**：三层知识图谱架构理念、症状-疾病映射表、GraphRAG增强诊断的设计模式

### 7.2 ClinicalPilot（2026）

- **定位**：面向临床试验的多Agent决策系统
- **特点**：Agent间具有协商与反馈机制，不仅仅是单向传递
- **本项目借鉴点**：Diagnosis Agent → Intake Agent的条件回环路由设计（信息不足时让Intake补充更多细节）

### 7.3 MediGenius

- **定位**：开源医疗AI助手平台
- **特点**：支持多模态输入（文本+影像），端到端临床决策支持
- **本项目借鉴点**：5个Agent的功能划分参考了MediGenius的"采集-诊断-治疗-编码-审计"全流程设计

### 7.4 LangGraph4j

- **仓库地址**：github.com/bsorrentino/langgraph4j
- **定位**：Python LangGraph的Java/JVM实现
- **特点**：将StateGraph、Node、Edge、ConditionalEdge等核心概念移植到JVM生态
- **版本**：本项目参考v1.8.11
- **本项目借鉴点**：Java实现中的Agent编排接口设计和状态流转思路

### 7.5 LangGraphGo

- **定位**：Go语言实现的LangGraph思想
- **特点**：利用Go的goroutine和channel实现Agent并发执行
- **本项目借鉴点**：Go版Pipeline中的Agent接口设计（`Agent interface { Name() string; Process(ctx, state) error }`）

- 集成GraphRAG知识图谱增强鉴别诊断，通过Neo4j多跳查询提升诊断证据链质量

- 提供Python/Java/Go三语言实现，API契约一致，结构化日志+错误降级
